# Greenlit — Supabase backend

Production-ready Postgres schema, RLS, private storage, and Edge Functions for Greenlit v2.

## Environments

Use three separate Supabase projects:

| Environment | Project name     | Purpose        |
|-------------|------------------|----------------|
| Development | `greenlit-dev`     | Local + preview |
| Staging     | `greenlit-staging` | Pre-prod QA     |
| Production  | `greenlit-prod`    | Live traffic    |

Each environment has its own project URL, anon key, and service role key. Configure them in:

- **Local:** `.env.local` (see root `.env.example`)
- **Vercel:** per-environment env vars in the dashboard

Never commit real keys. Never put the service role key or LLM API keys in the browser bundle.

## Schema overview

| Table            | Purpose                                      |
|------------------|----------------------------------------------|
| `shops`          | Repair shops (`id` UUID, `slug` for URLs)    |
| `shop_members`   | Links `auth.users` to shops                  |
| `intakes`        | One customer intake session                  |
| `intake_messages`| Shared conversation log (training data)      |
| `intake_media`   | Audio/video/photo in Storage; text inline    |
| `intake_ratings` | Mechanic outcome labels                      |
| `llm_call_log`   | Postgres-backed LLM rate-limit counter     |

Shops are addressed by **slug** in URLs (`/shop/demo-shop`). All foreign keys use UUID `shops.id`.

### `intakes` write authority

The **Edge Function** is authoritative for final intake fields. On `diagnostician_final`, the Edge Function updates in one transaction:

- `brief` (jsonb)
- `status = 'complete'`
- `urgency`
- `category`

The browser client never writes these fields.

## Row Level Security

RLS is enforced on all application tables. Summary:

| Table            | anon                          | authenticated shop member        |
|------------------|-------------------------------|----------------------------------|
| `shops`          | SELECT                        | SELECT                           |
| `shop_members`   | —                             | SELECT (same shop)               |
| `intakes`        | INSERT only                   | SELECT, UPDATE (own shop)        |
| `intake_messages`| INSERT (intake ≤ 30 min old)  | SELECT (own shop)                |
| `intake_media`   | INSERT (intake ≤ 30 min old)  | SELECT (own shop)                |
| `intake_ratings` | —                             | INSERT, SELECT, UPDATE (own shop)|
| `llm_call_log`   | — (service role only)         | —                                |

**Anon never SELECTs intake data.** The client caches the intake UUID in memory after creation. Reads go through the `llm-proxy` Edge Function (`get_intake` intent), which uses the service role server-side.

### 30-minute anon window (known limitation)

Anonymous users may INSERT into `intake_messages`, `intake_media`, and Storage only while the parent intake was created within the last 30 minutes. This bounds abuse but is not a cryptographic proof of ownership — anyone who knows an intake UUID within that window could append data. Acceptable for v2; tighten before public launch if needed.

### Storage

- Bucket: **`intake-media`** (private — no public reads)
- Path pattern: `{intake_id}/{media_id}.{ext}`
- Anon INSERT under `{intake_id}/` prefix during the 30-min window
- Reads: **signed URLs only** (5-min TTL), generated by the Edge Function
- Direct public URL fetch returns **403**

## LLM rate limiting

Rate limits are enforced in Postgres, not in-memory (Edge Function isolates are ephemeral).

Before each LLM call, the Edge Function counts rows in `llm_call_log`:

```sql
SELECT count(*) FROM llm_call_log
WHERE intake_id = $1
  AND called_at > now() - interval '5 minutes';
```

If count ≥ **20**, return **429**. After a successful LLM call, insert a row.

### Cleanup

Schedule periodic deletion of old log rows (pg_cron, Supabase scheduled function, or manual):

```sql
DELETE FROM llm_call_log
WHERE called_at < now() - interval '7 days';
```

## Edge Function: `llm-proxy`

Deploy:

```bash
supabase secrets set OPENAI_API_KEY=sk-...
supabase secrets set SUPABASE_SERVICE_ROLE_KEY=...
# Optional — defaults to OpenAI gpt-4o:
# supabase secrets set DIAGNOSTICIAN_API_URL=...
# supabase secrets set DIAGNOSTICIAN_API_KEY=...

supabase functions deploy llm-proxy
```

### Intents

| Intent                    | Description                                      |
|---------------------------|--------------------------------------------------|
| `interviewer`             | Question batch generation                        |
| `diagnostician_hypothesis`| Mid-intake hypothesis                            |
| `diagnostician_final`     | Final brief + authoritative `intakes` update     |
| `get_intake`              | Fetch intake + messages + media (signed URLs)    |
| `signed_url`              | 5-min signed URL for one `storage_path`          |

### LLM validation

Every LLM response is validated in the Edge Function (Zod via Deno). On validation failure: retry once with schema in the system message; second failure returns a structured error to the client. Unvalidated LLM output is never returned.

### Media passed to LLMs (v2)

No Whisper or vision in v2. Both models receive `media_summary` as metadata + text only:

```json
[
  { "kind": "audio", "duration_seconds": 12.4 },
  { "kind": "photo", "media_id": "..." },
  { "kind": "text", "text_content": "the whole freeform description" }
]
```

Do not invent transcripts or image descriptions.

## Prompt sync

System prompts live in `src/lib/ai/prompts/`. Before every production build, `scripts/sync-prompts.mjs` copies them to `supabase/functions/llm-proxy/prompts/`. The build fails if they differ.

```bash
node scripts/sync-prompts.mjs   # manual sync
npm run build                   # runs sync first
```
